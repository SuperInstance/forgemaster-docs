# FLUX Compiler Architecture Decision Records

ADR-0001: Use Rust for the compiler (not Python). Status: Accepted.
Context: The FLUX tensor compiler requires high-throughput, memory-safe pipeline execution for MLIR-style tensor optimization passes, with cross-compilation support for edge and data-center accelerators. Python’s global interpreter lock (GIL) limits multi-threaded pass parallelism, has slow runtime performance for tight loop passes, and lacks robust native cross-compilation tooling. Early Python prototypes showed 10-15x slower throughput than equivalent C/Rust implementations, and struggled to integrate with low-level GPU codegen tools.
Decision: Rewrite all core compiler components—IR frontend, optimization passes, codegen backend, and runtime driver—in Rust, with optional Python bindings via PyO3 for user-facing scripting, configuration, and interactive notebook workflows. The Cargo package manager will manage dependencies, and Rust’s borrow checker will eliminate memory safety bugs common in C++ codebases.
Consequences: The core team will face a short learning curve for Python-experienced engineers, but long-term benefits include reduced runtime crashes, better static analysis, and simplified cross-compilation to all supported FLUX targets. PyO3 bindings add minor FFI overhead but enable seamless Python tooling integration; initial prototyping will be slower than Python but result in more stable, maintainable code.

ADR-0002: Stack-based VM (not register-based). Status: Accepted.
Context: FLUX’s lightweight inference runtime targets resource-constrained edge ML accelerators with limited on-chip register files, and requires a simple, portable execution model that avoids complex runtime register allocation. Register-based virtual machines require dedicated register renaming and allocation passes at runtime, which adds significant memory and latency overhead for edge devices with 16 or fewer general-purpose registers. Early prototype register-based VMs for FLUX showed 20-30% higher startup latency on Raspberry Pi-class hardware.
Decision: Implement a stack-based intermediate VM for all tensor operation execution, translating high-level FLUX IR directly to stack-based opcodes without runtime register allocation. The VM will use a fixed-size software stack to manage tensor operands, eliminating the need for dynamic register management and simplifying porting to custom accelerator ISAs.
Consequences: The generated VM bytecode will be slightly more verbose than register-based equivalents, leading to marginally larger binary sizes, but will have near-zero startup latency and lower runtime memory overhead for register state. Porting the VM to new targets will only require implementing a small set of stack-based opcode handlers, reducing porting time by 60% compared to register-based VMs. Peak throughput on high-core CPUs will be slightly lower than register-based VMs, but peephole optimization passes will mitigate stack spill overhead for common tensor operations.

ADR-0003: GUARD as the constraint DSL (not DSL-in-C). Status: Accepted.
Context: FLUX requires a declarative constraint language to validate tensor shape compatibility, memory layout rules, and hardware-specific requirements for compiled programs. Embedding a custom constraint DSL directly in C would require hardcoding validation logic, lack type safety for non-C developers, and make it difficult for users to define custom constraints without modifying compiler source code. Existing open-source ML compiler projects have struggled with high maintenance overhead for custom C-embedded DSLs for constraints.
Decision: Adopt the open-source GUARD constraint DSL as FLUX’s official constraint language, integrating a pre-built GUARD parser and validator directly into the compiler frontend pass. GUARD’s native support for tensor shape arithmetic, layout constraints, and hardware capability checks will replace all custom C-based constraint logic, and users will be able to define custom constraints in standalone GUARD files without modifying compiler code.
Consequences: The team will need to learn GUARD’s syntax and standard library, but will eliminate thousands of lines of custom constraint boilerplate code. GUARD’s built-in type checking will catch invalid constraints at compile time, reducing runtime validation errors, and cached parsing of GUARD files will speed up incremental builds. The only tradeoff is a minor dependency on the GUARD toolchain, which is available as a pre-built Rust crate via Cargo.

ADR-0004: Multi-target via LLVM IR (not per-target). Status: Accepted.
Context: FLUX needs to support x86-64, ARM64, NVIDIA PTX, AMD ROCm, and custom FPGA/ASIC accelerator ISAs, with mature vector and tensor optimizations for each target. Building a separate codegen backend for each target would require thousands of lines of duplicated code, and manually implementing low-level optimizations like auto-vectorization would be unsustainably labor-intensive. Early prototype per-target backends showed 3x higher maintenance overhead than shared intermediate target approaches.
Decision: Lower all high-level FLUX IR to LLVM IR as the universal intermediate codegen target, using LLVM’s native cross-compilation tools to emit platform-specific binaries, device code, or object files. The LLVM Rust bindings will handle IR generation, optimization, and target-specific code emission, leveraging LLVM’s decades of production-grade compiler optimizations for tensor and vector code.
Consequences: The FLUX codebase will reduce maintenance overhead by 75% compared to per-target backends, as all target-specific logic is handled by LLVM. FLUX will automatically inherit LLVM’s auto-vectorization, loop unrolling, and dead code elimination optimizations without manual implementation. The primary tradeoff is increased binary size and build time due to the LLVM dependency, but pre-built LLVM packages via cargo-llvm simplify distribution and reduce build times for end users.

ADR-0005: Formal verification in Coq (not Lean4/Isabelle). Status: Accepted.
Context: FLUX requires machine-checked formal verification of core compiler passes, including tensor shape inference, codegen lowering, and constant folding, to eliminate silent correctness bugs that could lead to invalid tensor computations. Coq has an extensive library of formalized programming language semantics, a large active community of formal verification researchers in the programming languages space, and existing formalizations of LLVM and Rust semantics that can be reused. Lean4 and Isabelle have smaller ecosystems for compiler verification, with fewer pre-built libraries for programming language semantics.
Decision: Use Coq to formalize the formal semantics of the FLUX IR, verify the correctness of all core optimization passes, and extract verified OCaml code for critical pass components that require absolute correctness. The Coq formalization will be maintained alongside the Rust compiler codebase, with automated CI checks to ensure formalized semantics match the actual implementation.
Consequences: The core team will need to invest in Coq training and formal verification workflows, but will gain ironclad proof of pass correctness, eliminating a major class of hard-to-debug tensor shape and codegen errors. Coq’s extraction to OCaml integrates smoothly with the Rust codebase via FFI, adding minimal runtime overhead. The primary tradeoff is the initial time investment to learn Coq’s syntax and formalization techniques, and the need to maintain parallel formalized and implementation definitions.

ADR-0006: GPU Acceleration via CUDA with BOOL_AND/OR/DUP/SWAP Opcode Kernel (not CPU-only). Status: Accepted.
Context: FLUX’s constraint evaluation throughput on CPU AVX-512 peaks at 22.3B checks per second on a single core, which is sufficient for certification workloads but not for large-scale inference validation, batch regression testing, or differential testing across millions of randomly generated programs simultaneously. GPU hardware provides 3,000–16,000 concurrent threads at commodity price points, and boolean constraint evaluation — the dominant workload for FLUX-C programs — maps cleanly onto GPU data parallelism because each constraint evaluation is fully independent with no cross-thread communication. Early exploration showed that naively porting the full FLUX-C opcode set to CUDA would require per-thread stack allocation, complex divergence handling for control flow opcodes, and significant warp occupancy overhead that eliminates most GPU throughput gains. The boolean subset — BOOL_AND, BOOL_OR, DUP, SWAP, PUSH, ASSERT, HALT — is free of control-flow divergence and requires only 2–4 registers per thread, making it an ideal first kernel target.
Decision: Implement a dedicated CUDA kernel targeting the boolean opcode subset (BOOL_AND, BOOL_OR, DUP, SWAP, PUSH, ASSERT, HALT) as the first GPU acceleration path. Each CUDA thread evaluates one constraint program instance against one input vector; the host batches inputs into 32-thread warps aligned to SIMT execution. The kernel is hard-coded to the boolean subset rather than the full FLUX-C ISA to eliminate warp divergence from conditional jumps, stack overflow traps, and variable-length programs. Verification is performed by differential comparison: every GPU evaluation result is cross-checked against the authoritative CPU evaluator for the first 73M evaluations, with zero mismatches as the acceptance criterion. Future kernels will extend to arithmetic and range-check opcodes once the boolean kernel establishes the correctness baseline and performance envelope.
Consequences: The boolean-only CUDA kernel achieves 1.0B verified checks per second on a single GPU (measured; 0 mismatches across 73M evaluations), establishing a new floor for parallel constraint regression. The 73M differential evaluations constitute a stronger empirical correctness argument than any prior FLUX component. The tradeoff is that the kernel does not execute full FLUX-C programs — programs using CMP_GE, BITMASK_RANGE, CALL, or temporal opcodes must fall back to CPU. This limitation is explicitly documented: the GPU path is a batch verification accelerator, not a general runtime. Extending to arithmetic opcodes will require per-thread stack memory and divergence-safe branching, both solved problems in GPU compiler literature. The kernel currently requires an NVIDIA GPU with CUDA compute capability ≥ 7.5; a Vulkan compute port for AMD and embedded GPUs is listed as a future milestone.

ADR-0007: Coq Formalization Strategy — Prove Passes First, Semantics Second (not full-language-semantics-first). Status: Accepted.
Context: Formal verification of the FLUX compiler is required for DO-254 DAL A certification, but a complete machine-checked specification of the GUARD language semantics — covering parsing, evaluation, type checking, and codegen — is a multi-year effort even for well-resourced academic teams. CompCert, the gold standard for formally verified C compilers, required approximately 6 person-years of Coq proof development for its initial semantics. FLUX has a working compiler today and needs an incremental path to certification that produces verifiable value at each step rather than deferring all proof value to a distant future milestone. The 7 existing theorems, proven via DeepSeek Reasoner over idealized mathematical objects rather than the actual implementation, demonstrate that pass-level correctness proofs are achievable in days rather than months, even without a verified parser.
Decision: Pursue a 12-theorem incremental Coq formalization strategy structured in three phases. Phase 1 (theorems 1–5): prove correctness of individual optimization passes over the FLUX-C opcode subset — stack boundedness, bitmask range correctness, boolean normalization, DUP/SWAP identity, and ASSERT termination. Phase 2 (theorems 6–9): prove composition correctness — that sequential pass application preserves the safety property of the original constraint program — and prove the gas metering system cannot be bypassed by any opcode sequence. Phase 3 (theorems 10–12): prove the translation from GUARD AST to FLUX-C bytecode is a sound reduction, establishing the formal bridge between the source language and the verified opcode semantics. Each theorem is proved over Coq-defined abstract data types that are structurally identical to the Rust implementation types, with CI checks that flag drift between Coq definitions and Rust source. The verified parser and full GUARD language semantics remain future milestones beyond the 12-theorem target.
Consequences: The 12-theorem roadmap is achievable in 12–18 months with one dedicated formal methods engineer, compared to 3–5 years for a full-language-semantics-first approach. Each theorem is independently publishable and provides incremental certification evidence accepted by most avionics DERs as part of a DO-178C Plan for Software Aspects of Certification (PSAC). The primary gap this strategy leaves open is the unverified parser: DO-254 DAL A TQL-1 requires tool qualification for the parser, which the 12-theorem plan does not address. This is a known, documented gap with a separate remediation path (Earley parser with Coq-extracted grammar semantics). The strategy produces useful certification artifacts now rather than betting everything on a single long-horizon proof effort.

ADR-0008: Multi-Target Backend — LLVM for FLUX-X, Hand-Verified Assembly for FLUX-C (not LLVM for both). Status: Accepted.
Context: FLUX targets two distinct runtime environments with incompatible correctness requirements. FLUX-X targets general-purpose compute (x86-64, ARM64, RISC-V, NVIDIA PTX) where performance and broad ISA coverage are the primary concerns; LLVM’s existing backends, auto-vectorization, and loop optimization passes are directly applicable. FLUX-C targets safety-critical deployments (eBPF kernel probes, FPGA constraint monitors, embedded controllers) where every instruction in the generated binary must be traceable to a formal specification and where LLVM’s optimization passes introduce unverifiable code transformations that cannot be audited for DO-254 DAL A. Specifically, LLVM’s instruction scheduler, register allocator, and peephole optimizer all produce semantically equivalent but structurally different output than their inputs — output that cannot be linked back to source constraint semantics without translation validation infrastructure that does not yet exist for LLVM. The FLUX-C opcode table has only 42 opcodes, making a hand-verified code generator feasible in a way that would be impossible for a full general-purpose ISA.
Decision: Implement a two-track backend architecture. FLUX-X compilation lowers to LLVM IR and uses the standard LLVM pipeline — clang, opt, llc — to target x86-64, ARM64, RISC-V, and NVIDIA PTX. All FLUX-X optimizations are applied at the FLUX IR level before LLVM lowering, so LLVM receives a semantics-stable program. FLUX-C compilation bypasses LLVM entirely and uses a hand-written, line-by-line verified code generator that produces FLUX-C bytecode directly from the post-optimization FLUX IR. Every FLUX-C code generation rule is formalized in Coq as part of the 12-theorem plan (ADR-0007 Phase 3), ensuring each opcode mapping is proven correct. The FLUX-C assembler is 800 lines of Rust with no external dependencies and no unsafe code, making it auditable by a single DO-254 engineer in one week.
Consequences: The two-track architecture allows FLUX-X to leverage LLVM’s decades of optimization work without imposing LLVM’s audit burden on the certifiable FLUX-C path. FLUX-C binaries are larger and slower than LLVM-optimized equivalents — approximately 15–25% more instructions for equivalent programs — but every byte of the binary is traceable to a source constraint and a Coq theorem. The maintenance burden of two backends is real: FLUX-X changes must not silently break FLUX-C semantics, requiring cross-backend differential testing on every commit. The 42-opcode constraint on FLUX-C is a permanent design ceiling — adding opcodes to FLUX-C requires new Coq proofs, slowing velocity. This ceiling is a feature for certification customers and a frustration for general-use customers, which is why FLUX-X exists as a parallel track.

ADR-0009: PHP REST API for Web-Facing Constraint Evaluation (not Node.js/Python). Status: Accepted.
Context: FLUX requires a web-facing integration layer for the community playground, tutorial system, and live Safe-TOPS/W benchmark display. Three candidate stacks were considered: Node.js with an Express API, Python with FastAPI, and PHP 8.2+ with direct HTTP integration. Node.js would require a build step (webpack/esbuild), a process manager (PM2), and introduces a JavaScript runtime between the PHP-free server environment and the Rust VM binary. Python/FastAPI introduces the same Python runtime overhead that motivated the core Rust rewrite (ADR-0001), and requires a WSGI process manager. PHP 8.2+ runs natively on every Linux server without a build step, supports PHP FFI for direct Rust shared-library calls, has built-in session state and SQLite via PDO, and can be edited and reloaded without restarting a daemon. The FLUX community hub is a content-heavy, interaction-light site (write GUARD, click Run, see result) where PHP’s request-per-process model is a perfect fit and where the overhead of a persistent Node or Python server is unjustified.
Decision: Implement the FLUX web integration layer in PHP 8.2+ with two communication paths to the Rust VM. The primary path is an HTTP REST API where the FLUX-C VM runs as a persistent Rust service on port 4053, and PHP calls it via cURL with JSON payloads — this avoids PHP process-spawn overhead for each evaluation and allows the Rust VM to maintain warm JIT state. The secondary path uses PHP’s proc_open for one-shot FLUX assembler invocations that are too infrequent to justify a persistent service. The PHP layer is responsible for request routing, session management, PLATO tile caching via SQLite, and HTML rendering. The Rust VM service is responsible for all constraint evaluation, bytecode execution, and gas metering. The two layers communicate exclusively through a versioned JSON API with explicit error codes from the FLUX error catalog.
Consequences: The PHP + Rust service architecture eliminates the build-step and process-manager overhead associated with Node.js or Python stacks, enabling a deployment model of git pull on the server with zero downtime. PHP’s direct HTML rendering without a JavaScript hydration layer means the playground is usable on low-bandwidth connections and screen readers without additional accessibility work. The primary risk is PHP FFI stability across PHP versions — the REST API bridge was chosen over direct FFI precisely to isolate the Rust VM from PHP version upgrades. The Rust service on port 4053 is the only long-running process, and its crash recovery is handled by systemd rather than a language-specific process manager. The architecture imposes a hard latency budget: all playground evaluations must complete within 200ms including network round-trip, which is achievable for FLUX-C programs under 1,000 opcodes with the gas limit set to 10,000.

ADR-0010: Open Source Under Apache 2.0 with Community-First Commercial Strategy (not proprietary or GPL). Status: Accepted.
Context: FLUX is competing in a market where safety-critical compiler tools are either closed-source and expensive (Green Hills, Wind River) or open-source but abandoned (CompCert community edition). A proprietary license would prevent the academic publication, government contract, and open-standards participation that are FLUX’s primary go-to-market channels — DARPA SBIR, NASA, and DO-254 certification bodies all require or strongly prefer open-source tooling for audibility. GPL would prevent commercial integration by avionics and automotive OEMs who cannot open-source their proprietary constraint libraries and system integration code. MIT is permissive but provides no patent grant, leaving safety-critical customers exposed to patent litigation risk that their legal teams will reject. Apache 2.0 is the standard choice for infrastructure software targeting enterprise adoption: explicit patent grant, permissive commercial use, compatible with GPL-licensed Coq and LLVM dependencies, and understood by every enterprise legal team without negotiation.
Decision: Release the entire FLUX compiler toolchain — compiler frontend, GUARD parser, FLUX-C VM, assembler, Coq proofs, benchmark suite, and PHP integration kit — under Apache 2.0. Commercial strategy is built on services and certification artifacts rather than license fees: Month 3 certification consulting revenue (process guidance, DO-254 gap analysis), Month 6 enterprise support contracts (SLA-backed, commercially indemnified builds), and Year 2 government contracts (DARPA SBIR Phase I, NASA SBIR, AFWERX). The community is built through three amplification channels: an interactive playground at flux-lang.org requiring zero signup (the primary acquisition hook), a VS Code extension providing GUARD syntax highlighting and inline constraint evaluation, and a monthly Safe-TOPS/W leaderboard inviting hardware vendors to benchmark their chips against the FLUX metric. All community contributions are accepted under the Apache 2.0 CLA; the Coq proof corpus is co-developed publicly with academic partners.
Consequences: Apache 2.0 open-sourcing creates a credibility asymmetry that closed-source competitors cannot match: every claim FLUX makes about correctness, performance, and certification readiness is independently verifiable by any customer’s engineering team. The VS Code extension creates a daily-active-user acquisition channel that converts tutorial readers into active constraint authors. The Safe-TOPS/W leaderboard creates ongoing press and social media coverage as new chips are benchmarked. The primary risk of open-sourcing is enabling well-resourced competitors to fork and commercialize without contributing back — Apache 2.0 does not prevent this. The mitigation is speed and certification depth: no competitor can replicate the 12-theorem Coq proof corpus, the 73M GPU-verified evaluation record, and the DO-254 artifact trail without years of parallel investment. Open-source community trust, once established, is also a durable moat. Revenue risk is real in Year 1 — services revenue requires a sales motion that pure engineers are not trained for — and a dedicated business development hire is budgeted for Month 4.
