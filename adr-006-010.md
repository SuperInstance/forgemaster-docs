# Architecture Decision Records — FLUX System (ADR-006 to ADR-010)

## ADR-006: GPU Acceleration Strategy

### Context
The FLUX system needs to check constraints against millions of sensor inputs per second. CPU scalar achieves 5.2B/s but is single-threaded. GPU parallelism can process thousands of inputs simultaneously.

### Decision
Use CUDA kernels for batch constraint checking on NVIDIA GPUs. The kernel implements a minimal FLUX-C ISA with:
- `0x1D`: BITMASK_RANGE (range check)
- `0x1C`: CHECK_DOMAIN (bitwise domain membership)
- `0x26`: BOOL_AND (boolean conjunction)
- `0x27`: BOOL_OR (boolean disjunction)
- `0x28`: DUP (stack duplicate)
- `0x29`: SWAP (stack swap)
- `0x1B`: ASSERT (verify top of stack)
- `0x1A`: HALT

Compiled for SM 8.9 (Ada Lovelace, RTX 4050).

### Consequences
- **73M constraint evaluations verified with 0 mismatches** across 6 experiment types
- Throughput: 145-168M VM/s depending on bytecode complexity
- Power: 11-17W (RTX 4050 Laptop)
- Safe-TOPS/W: 39.5M certified checks/s per watt
- Limitation: 1-byte immediate values (range 0-255 per operand). Multi-byte ranges require CPU fallback or kernel extension.

## ADR-007: Coq Formalization Strategy

### Context
English proofs are not machine-checked. DO-254 DAL A requires verified tool qualification. DERs trust Coq (40 years, zero soundness bugs) more than human arguments.

### Decision
Mechanize the most important proofs in Coq, prioritized by certification impact:
1. **WCET theorems** (flux_wcet_coq.v): termination, determinism, no-infinite-loops, confluence — DONE (4 theorems)
2. **Galois connection** (flux_galois_coq.v): compile_correct, alpha_sound, alpha_complete, alpha_monotone — DONE (4 theorems)
3. **Composition** (flux_composition_coq.v): and_check_correct, or_check_correct, and_sound, and_n_correct — DONE (4 theorems)
4. **Bitmask functor**: Pending (category theory in Coq is harder)
5. **Safety confluence**: Pending (follows from WCET + composition)
6. **Memory safety**: Pending (requires modeling heap in Coq)

### Consequences
- 12 Coq theorems provide machine-checked evidence for DO-254 submission
- The Galois connection is the strongest possible compiler correctness theorem
- Remaining 18 English proofs need mechanization (estimated 6-9 months)
- Each mechanized proof strengthens the certification case

## ADR-008: Multi-Target Backend

### Context
FLUX constraints must run on diverse hardware: x86-64 (server), ARM Cortex-R (embedded), RISC-V (emerging), CUDA (GPU), WebAssembly (browser), FPGA (silicon).

### Decision
Two-ISA strategy:
- **FLUX-C**: 43 opcodes, Turing-incomplete, hand-verified backend. For safety-critical (DAL A/ASIL D).
- **FLUX-X**: 247 opcodes, general-purpose, LLVM backend. For non-safety or development.

The debate consensus (DeepSeek × Seed): "Don't fight LLVM. Ride it for FLUX-X."

### Consequences
- FLUX-C has trivial WCET analysis (proven in Coq)
- FLUX-X leverages LLVM's mature optimization and multi-target support
- TrustZone-style bridge between the two ISAs for mixed-criticality systems
- Certification effort focuses on FLUX-C only (FLUX-X inherits LLVM's qualification)

## ADR-009: PHP Integration Kit

### Context
Oracle1 builds websites for Cocapn. The FLUX system needs web-accessible constraint checking for demos and customer evaluation.

### Decision
Provide a drop-in PHP API endpoint:
- `POST /flux-api.php?action=check`: Accepts JSON constraints + values, returns pass/fail
- `GET /flux-api.php?action=benchmark`: Returns throughput statistics
- CORS headers for cross-origin access
- Zero PHP dependencies (pure implementation)

### Consequences
- Oracle1 can embed constraint checking in any PHP website
- The benchmark endpoint provides live Safe-TOPS/W measurements
- RESTful API serves as the basis for the future FLUX Certify portal
- Limitation: PHP is not safety-critical. This is for demos, not production certification.

## ADR-010: Community Building Strategy

### Context
FLUX needs users, contributors, and industry adoption. The adversarial debate revealed: "0 customers" is the biggest weakness. Open-source alone doesn't guarantee adoption.

### Decision
Multi-layered community strategy:
1. **Apache 2.0 license** — No patents, full freedom, no vendor lock-in
2. **FLUX Playground** — Browser-based, zero-install, instant gratification
3. **VS Code extension** — Syntax highlighting + autocomplete for .guard files
4. **Professional documentation** — Tutorials, man page, error codes, API docs
5. **University partnerships** — Teach constraint programming with FLUX
6. **Conference papers** — EMSOFT 2027 submission (47KB paper)
7. **Safe-TOPS/W open standard** — Invite industry to adopt the benchmark
8. **GitHub-first** — 7 focused repos with READMEs, CI, CONTRIBUTING guides

### Consequences
- The "why.html" page honestly addresses counter-arguments (builds trust)
- The playground demonstrates the system in 30 seconds (reduces friction)
- Apache 2.0 means no IP concerns for aerospace companies
- Risk: documentation without users is just noise. Need first customer within 6 months.
